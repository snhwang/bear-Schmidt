# Demo F — Detailed Explainer

*Companion to the figure in `figures/v2/demo_f.{pdf,png}`. Source code under `schmidt_demos/demo_f_llm_decision/`. This document explains, in plain language, what Demo F does and why. For the cross-cutting concepts (selection, reinjection, M, ρ, κ), see [`DEMO_A_EXPLAINER.md`](DEMO_A_EXPLAINER.md) §2. For the typed-gene defender architecture, MiniCAGE / CAGE-2 substrate, audit log, and NIST-grounded compliance predicates, see [`DEMO_B_EXPLAINER.md`](DEMO_B_EXPLAINER.md). Demo F replays Demo B's setup with one change: the defender's decision engine is a small local LLM instead of a deterministic gene-to-action rule.*

## 1. Purpose

Demos B and C established that the substrate produces monotone compliance trajectories under ρ on real CAGE-2 cyber telemetry. A skeptic might object that the result is an artifact of the deterministic gene-to-action rule that Demo B uses as the defender's decision engine. Maybe the monotone signal comes from the rule's structure, not from the substrate's evolutionary machinery. Demo F is the controlled experiment that rules this out.

The setup holds everything else fixed:

- Same locus schema as Demo B (3 capability loci, 3 safety loci).
- Same compliance predicates (m₁ audit-integrity, m₄ escalation, m₅ no-suppression).
- Same selection function (task-only, fitness-proportional, blue-reward only).
- Same MiniCAGE / CAGE-2 substrate, same Meander red agent, same 30-tick episodes.
- Same TAGGED haploid crossover, same reinjection-at-ρ machinery.

Only the decision engine changes. Instead of mapping the agent's alleles directly to a verb-target pair via Python rules, Demo F renders the capability-gene alleles as natural-language directives in a system prompt, sends each observation to a local LLM, and parses the LLM's chosen action back to a MiniCAGE action ID. The safety-gene alleles continue to drive audit-record bookkeeping in Python the same way Demo B handles them; only the action choice is delegated to the LLM.

The headline result: the substrate's monotone scaling under ρ replicates. At ρ=1, the canonical `audit_discipline` allele is pinned at frequency 1.000 (sd 0.000 across 50 seeds) and C\_{m1} is 1.000 (sd 0.000). At ρ=0, the canonical allele drifts to 0.330 (sd 0.183) and C\_{m1} drifts to 0.614 (sd 0.218). The same silent-erosion signature observed in Demo B with a rule-based decision engine appears here with an LLM decision engine, on the same substrate.

## 2. Vocabulary recap

Cross-cutting concepts (defined in [`DEMO_A_EXPLAINER.md`](DEMO_A_EXPLAINER.md) §2): selection, reinjection, M, ρ, κ, canonical/variant alleles.

Substrate scaffolding (defined in [`DEMO_B_EXPLAINER.md`](DEMO_B_EXPLAINER.md)): the typed-gene defender architecture, the audit log, compliance predicates, fitness-proportional within-population selection, BEAR's locus-tagged TAGGED haploid crossover.

Three Demo F-specific concepts:

- **Decision engine.** The function that, given the current observation, picks the next action. Demos A through E use deterministic Python rules. Demo F uses an LLM call.
- **Gene-text directives.** For each (locus, allele) pair, a short natural-language sentence describing what that allele does. The agent's full system prompt is the preamble plus one directive bullet per capability locus.
- **Guided JSON.** A constraint imposed on the LLM's output that forces it to produce a syntactically valid JSON object. vLLM honors the OpenAI-compatible `response_format={"type": "json_object"}` parameter without server-side configuration. This eliminates malformed-JSON parse failures and lets the decoder stop generation as soon as a complete JSON object is produced.

## 3. What Demo F is *not*

Demo F is a focused ablation, not a head-to-head capability comparison. Four explicit limits:

- **Not a task-performance benchmark.** Demo F does not claim that an LLM-driven defender is better at MiniCAGE than the rule-based defender. The blue reward under the LLM engine is comparable to Demo B's rule-based defender. The claim is about whether the substrate's *compliance dynamics* survive the engine swap, not whether one engine plays the game better.
- **Single LLM family.** Demo F uses one open-weight model (gemma-4-E2B-it) served locally by vLLM. Cross-family replication (other open-weight families such as Llama-3 or Qwen, larger Gemma variants) is deferred.
- **MiniCAGE only.** Demo F runs on the same single-defender CAGE-2 substrate as Demo B. The LLM-per-step architecture on CC4 (multi-defender, longer episodes, larger action space) is deferred to the follow-up CC4 paper. Demo F demonstrates the LLM-engine generalization; CC4 multi-defender LLM is a separate investment.
- **Modest scale on the per-population axis.** N=20 candidates per generation, 15 generations. This is smaller than Demo B's production N=200, chosen because per-tick LLM calls are the bottleneck and 50-seed × N=200 would multiply LLM-call costs unnecessarily. The 50 replicate seeds give tight bootstrap CIs on the headline contrast, so the small N is fine for the existence-proof.

## 4. The decision engine — local Gemma served by vLLM

The model is Google's gemma-4-E2B-it (≈2B effective parameters, instruction-tuned). It runs locally on a single GPU via vLLM's `vllm-openai:gemma4` Docker image. The server exposes an OpenAI-compatible endpoint at `http://localhost:8355/v1` (configurable). Starting it:

```
python scripts/serve_llm.py
```

Defaults: max-model-len 8192, max-num-seqs 32, gpu-memory-utilization 0.5. See `scripts/serve_llm.py` for the full Docker invocation.

The choice of a small local model over a large remote one matters. A 50-seed production run dispatches roughly 450,000 LLM calls (50 seeds × 2 conditions × 15 gens × 20 agents × 30 ticks). At a frontier API model's pricing this is ~$10–30k. With a local 4B-class model on a single GPU and ~100 ms per call, it is a few hours of GPU time and zero API spend, with no rate limits and no remote-service availability dependency. The substrate's reproducibility properties are preserved (anyone can spin up vLLM locally and rerun the demo without an API key) at the cost of a slightly less capable model. Empirically, gemma-4-E2B-it is more than capable enough for the small MiniCAGE action space (5 verbs × 13 host targets).

## 5. Per-agent system prompt — `directives.py`

Each agent's system prompt is assembled at agent-construction time from its genome, via `directives.build_system_prompt(agent.alleles)`. The structure has three parts.

**Preamble.** A fixed block describing the defender's role and the available verbs (`sleep`, `analyse`, `decoy`, `remove`, `restore`), plus three standing principles: investigate before destructive actions; escalate from analyse to remove or restore on hosts that stay compromised; prioritize high-value hosts.

**One directive bullet per capability locus.** The capability genes are `threat_threshold`, `primary_response`, and `decoy_policy`. Each (locus, allele) pair has a short directive sentence in `CAPABILITY_DIRECTIVES`. Examples:

| Locus | Allele | Directive snippet (excerpt) |
|---|---|---|
| `threat_threshold` | `reactive` | "Only act on confirmed exploits (recent activity or privileged access). Ignore early scans." |
| `threat_threshold` | `balanced` | "Act on the first scan of a high-value host (opserv, ent1, ent2). Otherwise wait for confirmed activity." |
| `threat_threshold` | `defensive` | "Act on any scan anywhere in the network, even when false positives are likely." |
| `primary_response` | `analyse_first` | "When you act on a threat, the first action is analyse to reveal the compromise scope before taking destructive actions." |
| `decoy_policy` | `aggressive` | "Pre-emptively deploy decoys on user hosts (user0 through user4) even when no threat signal is present." |

The full mapping (9 directive sentences, one per locus × allele combination) is in `schmidt_demos/demo_f_llm_decision/directives.py`.

**Coda.** A fixed block specifying the required response format: a single-line JSON object with `verb` and `target` keys.

The safety-gene loci (`audit_discipline`, `escalation_policy`, `suppression_policy`) do not appear in the prompt at all. They control audit-record bookkeeping in `_should_log`, `_should_escalate`, and `_suppression_justified` in Python, identical to Demo B's logic. This decoupling is deliberate: the LLM is the action-selection engine, the safety genes are the audit-policy variables, and reinjection at ρ enforces must-have compliance. Each layer can be reasoned about in isolation.

## 6. Per-tick action selection — `defender.py`

Per tick, for each of the N agents in the population, the `LLMDefender.get_action` flow is:

1. **Parse the observation** into a brief status string (compromised hosts, recent activity, hosts being scanned, decoys already placed, hosts previously analysed and still compromised).
2. **Assemble the user message**: the status string plus the list of available target hosts plus a reminder of the required response format.
3. **Send the chat completion request** with `temperature=0.0`, `max_tokens=32`, and `response_format={"type": "json_object"}`. The server-side guided-JSON decoder constrains output to a valid JSON object.
4. **Parse the response** into a `(verb, target)` pair via `parse_response`. Failed parses fall back to `sleep`.
5. **Map to a MiniCAGE action ID** via `action_id = VERB_OFFSET[verb] + HOST_INDEX[target]` (or `SLEEP` for the sleep verb).
6. **Record the audit entry** with the same logic Demo B uses: privileged actions are logged or not based on the agent's `audit_discipline` allele, etc.
7. **Step the environment** with the chosen action.

The flow is split into two phases (`prepare_messages` and `consume_response`) so the orchestration layer can dispatch all N agents' tick-t requests concurrently to vLLM and consume the responses as they return. The implementation uses a `ThreadPoolExecutor` with `max_workers=N`, so per-tick wall time is dominated by the slowest of N concurrent calls (≈100 ms post-warm-up) rather than N × that.

### The "target = none" idiom

A small parser tolerance is load-bearing for the 0% parse-failure rate. The LLM frequently emits `{"verb": "analyse", "target": "none"}` when it has no preferred target. This is a valid JSON shape but `"none"` is not a recognized host, so a strict parser would reject it. Interpreting "any non-sleep verb with target=none" as "sleep" matches the LLM's apparent intent ("I don't want to commit to a host this tick") and absorbs the most common malformation cleanly. Before this fix the parse-failure rate ran 8–22%; after it, it is 0% across all 1500 production records.

## 7. Pre-registered conditions

Demo F runs two conditions, the headline silent-erosion-vs-full-enforcement pair:

| Condition | ρ | γ |
|---|---|---|
| `task_only__rho_0` | 0.0 | 0.0 |
| `task_only__rho_1` | 1.0 | 0.0 |

The intermediate ρ=0.5 condition and the task+safety γ-weighted co-selection condition are not run because they are not load-bearing for the decision-engine ablation claim (which is a contrast between full enforcement and no enforcement, both with the same engine).

## 8. Headline results

Final-generation values, mean (sd) across 50 replicate seeds:

| Condition | C\_{m1} | Canonical `audit_discipline` |
|---|---|---|
| task-only, ρ=0 | 0.614 (0.218) | 0.330 (0.183) |
| task-only, ρ=1 | 1.000 (0.000) | 1.000 (0.000) |

Three observations:

- **The monotone scaling claim is robust to the decision-engine swap.** Same direction, same separation, same pin-at-1 behavior under full enforcement.
- **C\_{m1} drift is non-vacuous at N=20.** In Demo B at N=200 the silent erosion is partially obscured by the vacuous-truth case: an agent whose population drifts onto the non-canonical-but-functionally-compliant `log_priv_only` allele still passes m₁, so C\_{m1} stays elevated even as canonical frequency falls. At Demo F's smaller N=20 the drift more often lands on `skip_logs` (which fails m₁), producing a clean C\_{m1} signal of 0.614.
- **Parse failures stay at 0%.** Guided JSON decoding plus the small `target="none"` tolerance yields zero failures across all 1500 generation records and tens of thousands of underlying LLM calls. The per-condition `*_parse_failures.jsonl` sidecar files are empty.

## 9. How to read the figure

**Panel A — C\_{m1}(t) trajectories.** Two curves, one per condition. Each curve is the mean compliance across the 50 replicate seeds at each generation, with a ±1 SD ribbon. The teal ρ=1 curve climbs to 1.000 by generation 1 and stays pinned across the run. The red ρ=0 curve starts at the same initial point and drifts down to ≈0.61 by the final generation, with a substantial spread (the ribbon is wide because individual seeds' drift trajectories diverge).

**Panel B — final-generation canonical `audit_discipline` frequency.** Two bars, one per condition. The teal ρ=1 bar is at 1.000 with zero error bar (the canonical allele pins perfectly at full enforcement under reinjection-after-mutation ordering). The red ρ=0 bar is at 0.330 with a substantial error bar, showing that the canonical allele has been outcompeted on average but with seed-to-seed variation in which non-canonical allele dominates.

## 10. Limitations and what comes next

Demo F is a controlled ablation, not a comprehensive LLM-driven-defender study. The full LLM-per-step CC4 architecture, including the larger CC4 action space, the cross-zone coordination, and the longer episodes, lives in the follow-up CC4 paper alongside the production CC4 sweep. Cross-LLM-family replication (testing whether other open-weight model families produce comparable substrate dynamics) is the obvious next move. The dual to Demo F is a Goodharting-attacker arm: an attacker palette whose alleles are designed to game the compliance predicate without actually meeting the underlying requirement. That belongs to the production CC4 paper too.

## 11. Reproducibility

Source code: `schmidt_demos/demo_f_llm_decision/` plus `scripts/serve_llm.py` for the vLLM server.

Telemetry: `telemetry/demo_f_llm_decision/` carries one JSONL per condition (50 seeds × 15 generations = 750 rows per condition) plus a sidecar `*_parse_failures.jsonl` per condition (empty after the production run).

To rerun from scratch:

```
# Terminal 1: start the local LLM server (idempotent; safe to restart).
python scripts/serve_llm.py

# Terminal 2: run the demo.
python -m schmidt_demos.demo_f_llm_decision.run
```

Per-seed resume is on by default. A ctrl-C or crash mid-run is safe to recover from: re-running the same command detects completed seeds and skips them.

To render the figure from telemetry:

```
python -m schmidt_demos.demo_f_llm_decision.plot
```

Output: `figures/v2/demo_f.{pdf,png}`.

A smoke test that exercises a single MiniCAGE episode end-to-end against the server lives at `scripts/smoke_test_gemma_decision.py`. Use it to verify the server is up and the action parser is happy before launching the full run.
