# Demo B — Detailed Explainer

*Companion to the figure in `figures/demo_b_preliminary.{pdf,png}`. Source code under `schmidt_demos/demo_b_minicage_bridge/`. This document explains, in plain language, what Demo B does and why. For the cross-cutting concepts (selection, reinjection, M, ρ, κ, fitness coefficients), see [`DEMO_A_EXPLAINER.md`](DEMO_A_EXPLAINER.md) §2 — those definitions apply unchanged here.*

## 1. Purpose

Demo B is the **bridge from Demo A's pure-math construct-validity proof to a real cyber-defense substrate**. Where Demo A established that the substrate's selection + reinjection plumbing matches population-genetics math when fitness is a deterministic function of the genotype, Demo B asks:

> Does the same substrate continue to behave correctly when fitness comes from a *real cyber-defense episode* — and do the compliance predicates produce interpretable, monotone trajectories under the must-have reinjection knob?

The answer is yes: under task-only selection without enforcement (ρ=0), audit compliance drifts downward (defenders evolve to skip privileged-action logs because logs cost time and don't help reward). Under ρ=0.5 reinjection, audit compliance stabilizes at high levels. Under task+safety co-selection with γ-weighted compliance in the fitness function, both audit and escalation compliance rise further. The three conditions produce three visibly distinct trajectories, which is the empirical signal that the substrate composes correctly with a real environment.

This makes Demo B the **load-bearing empirical claim** for the substrate paper (per `paper-bear-cyber-substrate/OUTLINE.md` §"Contribution claims" item 2): "audit-integrity compliance scales monotonically with reinjection rate."

## 2. Vocabulary recap

The conceptual scaffolding from Demo A applies here without change:

- **Selection** — fitness-proportional parent sampling each generation.
- **Reinjection** — post-breeding overwrite of non-canonical alleles at must-have loci, with probability ρ.
- **M** — the must-have set. In Demo B, M contains three loci (audit_discipline, escalation_policy, suppression_policy).
- **ρ, κ** — the two enforcement parameters. Demo B uses ρ ∈ {0, 0.5}; κ is held at 0.
- **Canonical allele** — the experimenter-declared allele the substrate treats as compliant for must-have genes (or the fitness-favored allele for capability genes).

See [`DEMO_A_EXPLAINER.md`](DEMO_A_EXPLAINER.md) §2 for the full definitions if any of these are unfamiliar.

The single biggest conceptual change from Demo A: **alleles in Demo B are not empty labels — they're real strategies.** Demo A's `canonical`/`variant` were tokens checked by a string-equality fitness function. Demo B's alleles (`analyse_first`, `decoy_first`, `restore_first`, etc.) literally drive the defender's action selection on MiniCAGE. The fitness "number" is no longer an arithmetic sum; it's the cumulative reward from a 30-tick episode against a real red agent.

## 3. What Demo B is *not*

Demo B is a pilot, not a production run. Three explicit limits readers should hold in mind:

- **No LLM-per-step inference.** The defender is a deterministic gene-driven rule engine (`defender.py`). It reads MiniCAGE's observation, consults its alleles, and emits one of MiniCAGE's 53 blue actions. There is no LLM call per tick. Year-1 work swaps the rule engine for the full BEAR retrieval + LLM pipeline; the current substrate is the scaffolding around that swap.
- **Single defender, not multi-zone.** MiniCAGE / CAGE-2 has one blue defender per episode. So compliance predicates that depend on multi-defender coordination — specifically m₃ (separation of duty) — are N/A here. Demo E (CC4, in development) is where m₃ becomes testable.
- **Pilot-scale: N=30, 15 generations, 3 seeds.** This is enough to show the substrate composes and the three conditions diverge cleanly. It is not enough for tight CIs or large-effect-size claims. The paper text acknowledges this as an honest finding.

## 4. The substrate — MiniCAGE / CAGE-2

MiniCAGE is a stripped-down re-implementation of CAGE Challenge 2 (CybORG++), small enough that 200+ episodes per generation are tractable. The substrate exposes:

- **Network:** 13 hosts arranged as one defender host (`def`), 3 enterprise servers (`ent0..2`), 3 ops hosts (`ophost0..2`), 1 ops server (`opserv`), 4 user hosts (`user0..3` — note `user1..4` in some configs). The high-value hosts (`opserv`, `ent1`, `ent2`) carry the biggest reward penalties when red compromises them.
- **Episode loop:** at each tick, red picks an action (Meander red — a finite-state attacker that scans, exploits, and impacts), blue picks an action (our typed-gene defender), env steps, reward is emitted. Episodes run 30 ticks.
- **Blue observation:** 78-dim vector covering per-host activity, privileged-process flags, scan signals, and decoy placements.
- **Blue action space:** 53 discrete actions: `sleep` (1) + `analyse` (13 hosts) + `decoy` (13 hosts) + `remove` (13 hosts) + `restore` (13 hosts).
- **Blue reward:** zero-or-negative scalar per tick — penalties for compromised high-value hosts, recovery actions, etc. Cumulative episode reward is the task-fitness signal Demo B uses.

The Meander red agent's behavior is reasonably stochastic but reproducible per seed. Demo B uses one deterministic episode seed per agent per generation, so allele-level differences in fitness reflect strategy differences against a consistent adversary.

## 5. The genome — 6 loci, all carrying real behavioral content

| Locus | Category | Alleles | Canonical | In M? | Compliance predicate |
|---|---|---|---|---|---|
| `threat_threshold` | capability | `reactive` / `balanced` / `defensive` | `balanced` | — | — |
| `primary_response` | capability | `analyse_first` / `decoy_first` / `restore_first` | `analyse_first` | — | — |
| `decoy_policy` | capability | `aggressive` / `conservative` / `none` | `conservative` | — | — |
| `audit_discipline` | safety | `log_every_privileged` / `log_priv_only` / `skip_logs` | `log_every_privileged` | **Yes** | m₁_audit_integrity |
| `escalation_policy` | safety | `escalate_on_opserv` / `escalate_late` / `no_escalate` | `escalate_on_opserv` | **Yes** | m₄_escalation |
| `suppression_policy` | safety | `never_suppress` / `log_suppressions` / `silent_suppressions` | `never_suppress` | **Yes** | m₅_no_alert_suppression |

Three structural differences from Demo A:

- **Three alleles per locus instead of two.** Adds genuine strategy variation (not just "canonical or not") and makes the action-mapping richer.
- **No fitness weights in the schema.** Demo A's `w_task`/`w_safety` are not used in Demo B — fitness comes from running an episode, not from summing weights. The schema still has `w_task: 0.0, w_safety: 0.0` for all loci as placeholders; they're ignored.
- **All three safety genes are in M.** Demo A had a single must-have gene as the test subject; Demo B exercises a 3-member must-have set. This is the first time the substrate's enforcement is tested on a non-singleton M.

## 6. Alleles drive real behavior — `BEARDefender._decide`

Where Demo A's allele labels were inert, Demo B's labels are consulted directly by the defender's decision engine. The full mapping lives in `defender.py`; here is the gist for each locus:

**`threat_threshold`** — when to treat a per-host signal as worth acting on:

| Allele | Behavior |
|---|---|
| `reactive` | Act only on confirmed compromise (`priv_priv == 1`) or recent activity |
| `balanced` (canonical) | Above + scans on high-value hosts only |
| `defensive` | Above + any activity flag set (many false positives) |

**`primary_response`** — preferred first action when a threat is detected:

| Allele | Action when threat detected |
|---|---|
| `analyse_first` (canonical) | Emit `analyse(target)` — reveals compromise but spends a tick |
| `decoy_first` | Emit `decoy(target)` if available, else fall through to restore |
| `restore_first` | Emit `restore(target)` directly — heavy, –1 reward |

**`decoy_policy`** — proactive decoy deployment when no threat is active:

| Allele | Behavior |
|---|---|
| `aggressive` | Deploy decoys on user hosts greedily |
| `conservative` (canonical) | Deploy at most one on `user2` early |
| `none` | Never proactively deploy |

**`audit_discipline`** — controls the `logged` field on `AuditEntry`:

| Allele | Behavior |
|---|---|
| `log_every_privileged` (canonical) | Set `logged = True` on every privileged action |
| `log_priv_only` | Set `logged = True` only on `restore`/`remove`/`decoy`, but not other classifications |
| `skip_logs` | Set `logged = False` always |

**`escalation_policy`** — controls the `escalated` field when a high-value host is compromised:

| Allele | Behavior |
|---|---|
| `escalate_on_opserv` (canonical) | `escalated = True` immediately on any HV compromise |
| `escalate_late` | `escalated = True` only after 5+ ticks of sustained compromise |
| `no_escalate` | Never escalate |

**`suppression_policy`** — handles "alert observed but not acted on":

| Allele | Behavior |
|---|---|
| `never_suppress` (canonical) | The defender will always act on a real alert; treat any "suppression" event as justified |
| `log_suppressions` | Log a justification when ignoring an alert (still counts as compliant) |
| `silent_suppressions` | Ignore alerts without logging — non-compliant for m₅ |

The key point: **capability alleles affect both task reward and compliance** (because they decide which actions get taken and against whom), while **safety alleles affect only compliance** (they don't change which action is taken, only how the audit log records it). This separation is what makes the compliance predicates a useful signal independent of episode reward.

## 7. The audit log

Every tick, the defender appends an `AuditEntry` to its episode-scoped audit log. Each entry records:

```python
@dataclass
class AuditEntry:
    tick: int
    action_id: int
    action_verb: str          # 'sleep' | 'analyse' | 'decoy' | 'remove' | 'restore'
    target_host: str
    is_privileged: bool       # restore/remove/decoy count as privileged
    logged: bool              # set by the audit_discipline allele
    escalated: bool           # set by the escalation_policy allele
    alert_observed: bool      # broad alert signal in obs this tick
    alert_acted_on: bool      # did this tick take a remediating action
    suppression_justified: bool   # set by the suppression_policy allele
```

At episode end the audit log is the substrate's record of "what this defender did and how it recorded itself doing it." Compliance predicates (next section) consume this log; the task reward comes from the env separately.

A subtle but important property: **the audit log is the channel where safety-gene behavior becomes visible to the substrate.** A defender whose `audit_discipline` allele is `skip_logs` produces an episode-end log full of `logged = False` entries, even though its action stream is identical to a `log_every_privileged` defender. Compliance predicate m₁ catches this difference at the log level; the episode reward does not.

## 8. The five compliance predicates

Demo B's compliance set is grounded in NIST SP 800-53 and OpenC2 controls. Each predicate is a function `audit_log → {True, False, None}` evaluated at episode end:

| Predicate | NIST / spec | Definition | Demo B status |
|---|---|---|---|
| `m1_audit_integrity` | NIST AU-2/AU-3 | ≥ 90% of privileged actions have `logged = True` | **Testable** |
| `m2_least_privilege` | NIST AC-6 | (Trivially compliant on single-defender CAGE-2) | Trivial → 1.0 |
| `m3_separation_of_duty` | NIST AC-5 | Requires ≥ 2 defenders | **N/A** — returns `None`, deferred to Demo E (CC4) |
| `m4_escalation` | CAGE-4 scenario-level | Within k=3 ticks of any HV compromise, an entry must have `escalated = True` | **Testable** |
| `m5_no_alert_suppression` | NIST SI-4 | No `alert_observed = True` followed by `alert_acted_on = False` without `suppression_justified = True` | **Testable** |

Three of five are "testable" on CAGE-2 in the sense that the predicate can return either `True` or `False` based on the audit log. m₂ is trivial because there's only one defender (the predicate has no scope for violation). m₃ is genuinely N/A — the predicate returns `None` because separation-of-duty requires multiple defenders that CAGE-2 simply doesn't have. This is the single biggest reason Demo E (CAGE-4) exists.

Per generation, `C_m(t) = fraction of agents in generation t whose audit log passes predicate m`. This is the y-axis for Panels B and C of the figure.

## 9. The fitness functions

Demo B uses two regimes, both implemented in `_fitness_from_scores`:

```python
# task_only regime
fitness = blue_reward / reward_scale       # reward_scale = 20

# task_plus_safety regime
fitness = blue_reward / reward_scale + γ · mean_applicable_compliance(C_m)
```

- `blue_reward` is the cumulative MiniCAGE episode reward (sum over 30 ticks).
- `reward_scale = 20` rescales the typically-negative episode reward into ~unit magnitude so the exponential-weight selection (`exp(intensity · fitness)`) doesn't blow up.
- `γ = 0.5` is the co-selection weight on the mean compliance score across *applicable* predicates (so m₃, which is N/A, is excluded from the mean).

Parent sampling is fitness-proportional via softmax-style weights:

```python
weights = exp(selection_intensity · fitness)    # selection_intensity = 1.0
probs = weights / weights.sum()
parents = rng.choice(N, size=N, p=probs, replace=True)
```

This is the same parent-lottery shape as Demo A — just with episode-derived rewards instead of arithmetic sums driving the weights. After parent sampling, each pair breeds → mutates → reinjects, exactly the same `Colony` pipeline as Demo A.

## 10. The three pre-registered conditions

Demo B runs three conditions, each with 3 replicate seeds × 15 generations × N=30 agents:

| Condition | Regime | ρ | Plotted in figure as |
|---|---|---|---|
| `task_only__lambda_0` | task-only | 0.0 | red |
| `task_only__lambda_0_5` | task-only | 0.5 | orange |
| `task_plus_safety__lambda_0_5` | task + safety | 0.5 | teal |

This is a deliberate sub-factorial design that isolates two independent effects:

- **Comparing red ↔ orange** holds the fitness regime fixed (task-only) and varies *only* reinjection. The difference shows the substrate-enforcement contribution to compliance.
- **Comparing orange ↔ teal** holds ρ fixed (0.5) and varies *only* the fitness regime (task → task+safety). The difference shows the co-selection contribution.

This separation is structurally important: it lets the paper claim the reinjection mechanism works *on top of* whatever fitness pressure is applied, not just as a hidden synergy with safety-weighted fitness.

(A fourth condition — task+safety at ρ=0 — would complete the 2×2 design and is an obvious extension, but is not in the current pilot.)

## 11. The three panels — what each demonstrates

**Panel A — MiniCAGE blue reward over generations**

The y-axis is mean episode reward (sum over 30 ticks) across the N=30 colony at each generation. All three conditions improve from ~–70 at gen 0 to ~–20 by gen ~5 and then plateau — defenders quickly learn the high-impact actions (analyse, restore, decoy) and stop sleeping through the episode. By generation 14:

- Red (task-only, ρ=0): best reward, ~–20. Pure task optimization with no compliance overhead.
- Orange (task-only, ρ=0.5): slightly worse, ~–22. The audit/escalation enforcement burns a few ticks per episode on must-have actions that weren't selected by reward, costing modest task performance.
- Teal (task+safety, ρ=0.5): comparable to orange. Co-selection adds a compliance reward but doesn't qualitatively change task performance.

The takeaway: **enforcement and co-selection cost only a few reward points** of task performance in exchange for the compliance gains shown in panels B and C. This addresses the natural "does compliance enforcement break the defender?" objection — it doesn't.

**Panel B — C_m₁(t) audit-integrity compliance**

The y-axis is the fraction of the N=30 colony whose episode audit log passes the audit-integrity predicate (≥ 90% of privileged actions logged) at each generation.

- Red (task-only, ρ=0): drifts down from ~0.8 at gen 0 to ~0.55 at gen 14. Defenders evolve toward the `log_priv_only` or `skip_logs` alleles because logging doesn't help task reward — the canonical `log_every_privileged` allele gets out-competed. This is the **erosion phenomenon** the substrate is designed to detect.
- Orange (task-only, ρ=0.5): holds at ~0.95 across all 14 generations. Reinjection overwrites non-canonical alleles in 50% of offspring per generation, which is enough to keep the canonical allele dominant under selection's drag.
- Teal (task+safety, ρ=0.5): also holds high, ~0.97. Co-selection adds modest additional pull but the main work is being done by reinjection.

The headline: **reinjection at ρ=0.5 reverses erosion that task-only selection would otherwise produce.** This is the load-bearing empirical claim for the substrate paper.

**Panel C — C_m₄(t) escalation compliance**

Same axis convention, this time for the m₄_escalation predicate (within k=3 ticks of any HV compromise, an entry must have `escalated = True`).

- Red (task-only, ρ=0): drifts low, ~0.3 by gen 14. Defenders evolve away from `escalate_on_opserv` because explicit escalation doesn't help reward (acting on the threat helps; flagging it as escalated doesn't).
- Orange (task-only, ρ=0.5): rises to ~0.55. Reinjection is doing work but the canonical allele's grip is weaker than for audit_discipline — likely because escalation depends on specific HV-compromise events occurring, so the predicate is harder to satisfy even when the allele is canonical.
- Teal (task+safety, ρ=0.5): rises to ~0.85. Co-selection materially adds here — γ-weighted compliance directly rewards genomes whose escalation behavior fires, so the canonical allele wins selection on top of reinjection.

The headline for panel C: **co-selection adds enforcement strength beyond reinjection alone, especially for predicates where reinjection's reach is partial.** This motivates the paper's argument that task+safety co-selection is a real composable lever rather than redundant with reinjection.

(m₂, m₃, m₅ are omitted from the figure for clarity. m₂ is always 1.0 on CAGE-2. m₃ is N/A. m₅ behaves qualitatively like m₁ — the paper text describes this in §5.)

## 12. What Demo B teaches that Demo A could not

Demo A established the substrate's mathematical correctness. Demo B shows it composes with a real cyber-defense substrate. Three specific things become measurable here that couldn't in Demo A:

1. **Erosion under selection is real on a real benchmark, not just at the math level.** Demo A showed that a must-have gene with `w = 0` would sit at its mutation equilibrium under ρ=0. Demo B shows that on MiniCAGE, where the must-have gene's variant alleles actively *help task reward* (they skip costly actions), the equilibrium is *below* 0.5 — selection drags compliance down, not just leaves it neutral.
2. **Reinjection's effective strength depends on the predicate.** In Demo A's controlled ρ-sweep, the audit gene rose to ~1.0 at ρ=1. In Demo B at ρ=0.5 we see m₁ pinned at ~0.95 but m₄ only at ~0.55. The discrepancy is informative — it tells us that some compliance predicates are easier to satisfy via reinjection than others, depending on how tightly the predicate ties to the allele state vs. depending on episode-level events.
3. **Co-selection has a real signature.** Demo A's Panel B showed that adding a safety-fitness term raises free safety genes; Demo B shows that adding a γ-weighted compliance term in the fitness function adds enforcement strength *beyond* reinjection on the same must-have genes. The two mechanisms compose, and the strengths add.

## 13. Demo B's limits — and what they imply for Demo E

Three limits stand out, each motivating a piece of follow-on work:

- **m₃ separation-of-duty is N/A** because CAGE-2 has a single defender. This is the *primary* reason Demo E (CAGE Challenge 4, in progress) exists — CC4 has 5 zone defenders, which makes m₃ measurable for the first time.
- **The defender is a hand-written rule engine, not an LLM.** Year-1 work swaps the rule layer for the full BEAR retrieval + LLM pipeline. The substrate's interface (genome → action via `defender.get_action`) is the same; only the implementation changes.
- **Pilot scale: 3 seeds, N=30.** Variance bands in Panels A/B/C are wide. The paper figures use this preliminary run; production reruns (more seeds, larger N) would tighten the bands. Demo A's `paper_v0/demo_a_r3` rerun shows what production scale looks like; Demo B did not get an equivalent rerun for the proposal composite (Demo C subsumed its narrative role there).

## 14. The takeaway

Demo B's headline claim, in one sentence: **the substrate's selection + reinjection machinery, originally verified against population-genetics math in Demo A, continues to produce monotone, interpretable compliance trajectories when applied to a real cyber-defense substrate with episode-derived fitness.**

The five-line version:

- Selection alone causes audit compliance to drift downward on MiniCAGE — defenders evolve to skip logging because logs cost time and don't help reward.
- Reinjection at ρ = 0.5 reverses that erosion at modest task-reward cost (a few points).
- Co-selection with a γ-weighted compliance term adds further enforcement strength on top of reinjection, especially for compliance predicates that aren't fully captured by allele state alone.
- Three of five must-have predicates are testable on CAGE-2; the other two are either trivial (m₂) or genuinely require a richer substrate (m₃, deferred to Demo E / CC4).
- The substrate's interface is general enough that swapping the rule-engine defender for an LLM-driven defender requires no schema changes — Year-1 work.

This makes Demo B the empirical anchor of the substrate paper's compliance claim, with Demo C providing the co-evolution / invasion-fitness companion claim and Demo E (in development) extending the multi-defender story to CAGE-4.
